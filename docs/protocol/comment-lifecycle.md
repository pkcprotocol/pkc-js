# Comment Lifecycle

<!-- Note: "subplebbit" is being renamed to "community" — see RENAMING_GUIDE.md -->

## Summary

A comment goes through three distinct representations during its lifecycle. The author publishes a `CommentPubsubMessage`, the subplebbit stores it as an immutable `CommentIpfs` on IPFS, and the subplebbit periodically updates its mutable metadata via `CommentUpdate`. At runtime, the `Comment` class merges `CommentIpfs` + `CommentUpdate` into one object.

## Key Types

| Type | Source | Created by | Signed by | Mutable? | Access via `comment.raw` |
|------|--------|-----------|-----------|----------|--------------------------|
| `CommentPubsubMessagePublication` | `src/publications/comment/schema.ts` | Author | Author | No — ephemeral, only exists during pubsub exchange | `comment.raw.pubsubMessageToPublish` |
| `CommentIpfs` | `src/publications/comment/schema.ts` | Subplebbit (from author's pubsub message) | Author (original signature preserved) | **No — permanent on IPFS with CID** | `comment.raw.comment` |
| `CommentUpdate` | `src/publications/comment/schema.ts` | Subplebbit | Subplebbit owner | **Yes — regenerated when state changes** | `comment.raw.commentUpdate` |
| `Comment` (class) | `src/publications/comment/comment.ts` | Client-side merge | N/A (composed) | Runtime-only, not stored | The instance itself (merged view) |

## Invariants

- `CommentIpfs` is **immutable** once stored on IPFS. It gets a CID that never changes.
- `CommentUpdate` is the **only** way to add mutable state (votes, mod actions, edit merges) to a comment.
- `CommentUpdate` is signed by the **subplebbit owner**, not the author.
- A comment's bytes size during publication is limited to **40kb**.
- The `Comment` class merges immutable + mutable data. `comment.raw.comment` = original `CommentIpfs`, `comment.raw.commentUpdate` = latest `CommentUpdate`.

## Data Flow

```
1. Author → createComment({ content, title, ... })
2. Author → comment.publish()
   └─ Sends encrypted ChallengeRequestMessage to subplebbit's pubsub topic
3. Subplebbit → Sends ChallengeMessage (e.g., "solve 2+3")
4. Author → Sends ChallengeAnswerMessage ("5")
5. Subplebbit → Verifies answer
   ├─ On success: Stores CommentIpfs to IPFS → assigns CID
   │   └─ Sends ChallengeVerificationMessage with { comment: CommentIpfs, commentUpdate }
   └─ On failure: Sends ChallengeVerificationMessage with challengeSuccess=false
6. Subplebbit → Over time, regenerates CommentUpdate as votes/replies accumulate
7. Clients → Fetch CommentIpfs by CID + latest CommentUpdate from subplebbit → merge
```

## CommentIpfs Fields

Extends `CommentPubsubMessage` with subplebbit-added fields:
- `depth` — 0 for posts, 1+ for replies
- `thumbnailUrl`, `thumbnailUrlWidth`, `thumbnailUrlHeight` — generated by subplebbit for link posts
- `previousCid` — CID of author's previous comment (linked list per author)
- `pseudonymityMode` — `"per-post"` | `"per-reply"` | `"per-author"` | undefined

## CommentUpdate Fields

- `cid` — CID of the CommentIpfs this update is for
- `upvoteCount`, `downvoteCount`, `replyCount`, `childCount`
- `edit` — most recent author edit (validated separately, must match author's publicKey)
- `flairs`, `spoiler`, `nsfw` — mod-set overrides
- `pinned`, `locked`, `archived`, `removed`, `approved` — mod actions
- `reason` — reason for mod action
- `author.subplebbit` — per-community author stats (postScore, replyScore, banExpiresAt)
- `lastChildCid`, `lastReplyTimestamp` — for efficient update checking
- `replies` — pre-loaded first page of replies
- `signature` — subplebbit owner's signature
- `updatedAt`, `protocolVersion`

## Comment States

- `CommentState`: `"stopped"` | `"publishing"` | `"updating"`
- `PublicationPublishingState`: `"stopped"` → `"publishing-challenge-request"` → `"waiting-challenge"` → `"publishing-challenge-answer"` → `"waiting-challenge-verification"` → `"succeeded"` / `"failed"`
- `CommentUpdatingState`: `"stopped"` | `"resolving-author-name"` | `"fetching-ipfs"` | `"fetching-update-ipfs"` | `"resolving-community-name"` | `"fetching-subplebbit-ipns"` | `"fetching-subplebbit-ipfs"` | `"succeeded"` | `"failed"` | `"waiting-retry"`

## Common Mistakes

- Treating `CommentIpfs` as mutable — it is permanent. Only `CommentUpdate` changes.
- Confusing who signs what — `CommentIpfs` keeps the author's original signature; `CommentUpdate` is signed by the subplebbit owner.
- Forgetting that `replyCount` includes both direct and indirect children, while `childCount` (optional) is only direct children.
- Adding fields to `CommentIpfs` that should be in `CommentUpdate` — if the data can change over time, it belongs in `CommentUpdate`.

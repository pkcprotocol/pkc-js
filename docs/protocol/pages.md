# Pages and Pagination

<!-- Note: "subplebbit" is being renamed to "community" — see RENAMING_GUIDE.md -->

## Summary

Pages are sorted, paginated views of comments. They are **ephemeral** — regenerated by the subplebbit whenever comments, votes, or mod actions change. Each page contains an array of `{ comment: CommentIpfs, commentUpdate: CommentUpdate }` pairs and an optional `nextCid` for the next page.

## Page Types

| Type | Where | Schema | Contains |
|------|-------|--------|----------|
| `PostsPages` | `subplebbit.posts` | `PostsPagesIpfsSchema` | Top-level posts (depth=0) in a community |
| `RepliesPages` | `commentUpdate.replies` | `RepliesPagesIpfsSchema` | Replies to a specific comment |
| `ModQueuePages` | Internal to subplebbit | `ModQueuePagesIpfsSchema` | Comments pending moderator approval |

All schemas defined in `src/pages/schema.ts`.

## Structure

```typescript
// PostsPagesIpfs / RepliesPagesIpfs
{
  pages: {
    [sortName: string]: PageIpfs  // Pre-loaded first page per sort type
  },
  pageCids?: {
    [sortName: string]: string    // CID of the first page for additional sort types
  }
}

// PageIpfs
{
  comments: Array<{
    comment: CommentIpfs,         // Immutable comment data
    commentUpdate: CommentUpdate  // Latest mutable metadata
  }>,
  nextCid?: string                // CID of next page (pagination)
}
```

## Sort Types

### Posts (`PostSortNameSchema`)

`"hot"` | `"new"` | `"topHour"` | `"topDay"` | `"topWeek"` | `"topMonth"` | `"topYear"` | `"topAll"` | `"active"`

### Replies (`ReplySortNameSchema`)

`"best"` | `"new"` | `"old"` | `"newFlat"` | `"oldFlat"`

Both schemas also accept `z.string().min(1)` for forward compatibility with custom sort types.

## How Pages Work

1. Subplebbit generates pages from its comment database (see `src/runtime/node/subplebbit/page-generator.ts`)
2. First page of each sort is embedded directly in `pages` (pre-loaded — no extra fetch needed)
3. If more pages exist, `pageCids` provides the CID to fetch the next page
4. Each subsequent page has its own `nextCid` for further pagination
5. `pageCids` may be omitted if all items fit in the pre-loaded first page

## Runtime Types

In the `Comment` class at runtime, pages contain `CommentWithinRepliesPostsPageJson` objects which add runtime fields:
- `shortCid` — truncated CID for display
- `shortCommunityAddress` — truncated community address
- `author` — with `shortAddress` and `subplebbit` stats
- `deleted` — merged from edit
- `replies` — nested replies pages

## Invariants

- Pages are **ephemeral** — they are regenerated, not permanent. Do not rely on page CIDs being stable.
- Each page entry contains both `CommentIpfs` and `CommentUpdate` — clients don't need a separate fetch.
- `pageCids` is optional — if all items fit in one page, it may be omitted.
- ModQueuePages use `CommentUpdateForChallengeVerification` instead of full `CommentUpdate`.
- Page verification: every comment in a page must belong to the correct subplebbit and have valid signatures.

## Common Mistakes

- Treating page CIDs as permanent — pages are regenerated when state changes.
- Fetching pages without verifying signatures — use `verifyPage()` from `src/signer/signatures.ts`.
- Confusing `pages` (pre-loaded first page) with `pageCids` (CIDs to fetch more pages).

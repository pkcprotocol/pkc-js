# Data Permanence

<!-- Note: "subplebbit" is being renamed to "community" — see RENAMING_GUIDE.md -->

## Summary

The protocol has two categories of data: **permanent** (content-addressed on IPFS, immutable CID) and **ephemeral** (regenerated by the subplebbit, mutable). Understanding which data is permanent vs ephemeral is critical for making correct assumptions about storage, caching, and mutability.

## Permanent Data

Only two types of data persist forever and are never discarded:

| Data | Created By | Identified By | Contents |
|------|-----------|--------------|----------|
| `CommentIpfs` | Subplebbit (from author's publication) | CID (v0) | Author's comment content, signature, metadata |
| `CommentEdit` | Author | Published via challenge flow | Author's edit request (content changes, delete flag, flairs) |

These are content-addressed on IPFS — once published, the content at a given CID never changes, and the subplebbit retains them indefinitely unless they get purged.

## Ephemeral Data (Regenerated)

These are mutable — the subplebbit regenerates and republishes them as state changes.

| Data | Updated By | Identified By | Frequency |
|------|-----------|--------------|-----------|
| `CommentUpdate` | Subplebbit owner | Fetched via subplebbit's postUpdates mechanism | When votes, replies, or mod actions change |
| `SubplebbitIpfs` | Subplebbit owner | IPNS name (mutable pointer) | When any community state changes |
| Pages | Subplebbit owner | CIDs referenced in SubplebbitIpfs or CommentUpdate | When comments/votes change sort order |

Pages are regenerated with **new CIDs** on each update. Old page CIDs are unpinned and discarded by the subplebbit — they are not kept around.

## IPFS vs IPNS

- **IPFS** (InterPlanetary File System): Content-addressed storage. `CID → data`. Immutable.
- **IPNS** (InterPlanetary Name System): Mutable pointer. `IPNS name → CID`. The IPNS name is derived from the subplebbit's Ed25519 public key. Points to the latest `SubplebbitIpfs` CID.

```
IPNS name (stable) ──points to──> CID of SubplebbitIpfs (changes over time)
                                   └── pageCids ──> CID of PageIpfs (changes over time)
                                   └── posts.pages ──> embedded first pages
```

## postUpdates (CommentUpdate Distribution)

CommentUpdates are distributed via a bucket-based mechanism:
- Post CIDs are grouped into time-based buckets (by `postUpdatesBucket` timestamp)
- Clients only need to check the bucket for their post's age range
- Avoids fetching updates for all comments — only check relevant buckets

## A Comment's Visible State

At runtime, a comment's visible state is the merge of its permanent and ephemeral parts:

```
Visible Comment = CommentIpfs (permanent) + CommentUpdate (ephemeral)
```

- `content`, `title`, `link`, `author.name`, `timestamp`, `depth` → from CommentIpfs (permanent)
- `upvoteCount`, `downvoteCount`, `replyCount`, `pinned`, `removed`, `edit` → from CommentUpdate (ephemeral)

## HTTP Router Announcements

- Each HTTP router keeps provider announcements for only **24 hours**.
- This means content that hasn't been re-announced within 24h may become unfindable via HTTP routers.

## Invariants

- **Never modify a CommentIpfs** — once it has a CID, the content is immutable.
- **CommentUpdate is the only mechanism** for mutable comment state (votes, mod actions, author edits).
- **SubplebbitIpfs is a snapshot** — it represents the community state at a point in time, not a persistent record.
- **Page CIDs are not stable** — don't cache them long-term or rely on them as permanent identifiers.

## Common Mistakes

- Treating pages as permanent — they are regenerated with new CIDs.
- Trying to modify CommentIpfs after publication — any changes must go through CommentUpdate.
- Caching SubplebbitIpfs CIDs — the IPNS name is stable, but the CID it points to changes.
- Confusing "permanent data" with "always available" — IPFS data can become unavailable if no nodes pin it, even though its CID is technically immutable.
